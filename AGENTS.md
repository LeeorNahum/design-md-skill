# AGENTS.md

Rules for editing the **design-md** skill. User-facing guidance lives in `SKILL.md`. `README.md` is the human skim layer.

## Shape

A slim skill that redirects to one vendored reference. The skill teaches how to create, update, use, and validate a `DESIGN.md`; the spec itself is not restated in `SKILL.md`.

| File | Role |
| --- | --- |
| `SKILL.md` | Trigger, the three modes, sync rule, CLI validation, maintenance note |
| `references/spec.md` | Vendored upstream spec, generated by the sync script |
| `scripts/sync.mjs` | Fetches `docs/spec.md` from the upstream repo into `references/spec.md` |
| `.github/workflows/sync-upstream.yml` | Weekly and on-demand sync, commits only on change |
| `README.md` | Short human summary |

## Editing

- Bump `metadata.version` with semver in the same change whenever behavior changes.
- Never hand-edit `references/spec.md`. It is generated and overwritten by the sync script. Change it upstream or change the sync script.
- The upstream source of truth is `docs/spec.md` in `google-labs-code/design.md`. Keep the link and the sync target pointing there.
- Keep `SKILL.md` slim. Do not copy spec content into it; point to `references/spec.md`.
- Enforce `lint` as the validation gate. Keep the Tailwind export and diff optional, never mandated.
- **No em dashes.** Use commas, periods, parentheses, or "to".
- **Capitalized bullets.** Start every bullet with a capital letter.

## Sync provenance

The workflow records the exact upstream commit in each sync commit message. Do not add a commit SHA inside `references/spec.md`; provenance stays in git history so the file changes only when the spec content changes.

## Before finishing

- `references/spec.md` is present and matches what the sync script produces.
- `metadata.version` bumped if behavior changed.
